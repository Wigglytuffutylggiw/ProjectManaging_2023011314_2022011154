## 2. 分支策略

### 分支结构
```
main          # 主分支，保护分支，仅接受合并
  └── dev     # 开发集成分支
        ├── feature-wbs        # WBS功能分支
        ├── feature-schedule   # 进度计划分支
        └── feature-summary    # 总结报告分支
```

### 分支命名约定
- `main`：主分支，存放稳定版本
- `dev`：开发分支，集成分支
- `feature-<功能名>`：功能特性分支
- `hotfix-<问题描述>`：紧急修复分支（本项目暂不使用）

### 分支管理规则
1. **main 分支**：
   - 仅接受来自 dev 分支的合并
   - 由项目经理负责合并
   - 禁止直接提交

2. **dev 分支**：
   - 集成分支，整合各 feature 分支
   - 由配置管理员负责管理

3. **feature 分支**：
   - 基于 dev 创建
   - 完成后合并到 dev
   - 由各功能负责人管理

## 3. 提交命名约定

### 提交类型
| 类型 | 说明 | 示例 |
|---|---|---|
| feat | 新功能/新交付物 | feat: 完成WBS分解 |
| fix | 修复问题 | fix: 修正进度计划日期 |
| docs | 文档更新 | docs: 更新README |
| refactor | 重构 | refactor: 优化目录结构 |
| test | 测试相关 | test: 添加测试用例 |
| init | 初始化 | init: 初始化项目结构 |
| merge | 分支合并 | merge: 合并feature-wbs到dev |

### 提交信息格式
```
<类型>: <简短描述>

<详细说明（可选）>
```

## 4. 合并规则

### 合并流程
1. 功能开发完成后，提交 Pull Request
2. 至少一名团队成员审核
3. 审核通过后由配置管理员合并到 dev
4. 阶段性完成后由项目经理合并到 main

### 冲突处理规则
1. 冲突必须在合并前解决
2. 冲突解决由相关责任人协商处理
3. 冲突解决后必须保留处理记录
4. 重大冲突需团队共同讨论决定

## 5. 版本留痕

### 版本标记策略
- 使用语义化版本号：v1.0.0
- 主版本号：重大变更
- 次版本号：新增功能
- 修订号：修复问题

### 提交记录要求
- 每次提交必须有清晰的描述
- 关键交付物完成必须单独提交
- 保留完整的分支合并历史

### 文档版本管理
- 文档版本与代码版本保持一致
- 文档更新必须同步提交
- 保留文档变更历史

## 6. 配置管理员职责

1. 维护分支结构和合并规则
2. 监控提交质量和命名规范
3. 处理合并冲突
4. 管理版本标记
5. 确保仓库安全和完整性

## 7. 协作流程

```
创建feature分支 → 开发 → 提交 → PR审核 → 合并到dev → 测试验证 → 合并到main
```
