## 2. WBS 分解思路

### 2.1 分解原则
1. **结果导向**：所有条目均为可交付成果
2. **三级分解**：一级为成果域，二级为子成果，三级为最小成果单元
3. **完整性**：覆盖项目全部工作范围
4. **独立性**：各交付物之间相对独立

### 2.2 分解结构
- **项目管理成果**：启动成果、计划成果、控制成果
- **业务功能成果**：活动管理、报名管理、签到管理
- **技术实现成果**：架构设计、开发、部署
- **测试验证成果**：测试计划、测试执行

### 2.3 分解过程
1. 首先确定四大成果域作为一级条目
2. 每个成果域下分解子成果作为二级条目
3. 每个子成果进一步分解为具体交付物作为三级条目

## 5. 冲突解决过程记录

### 5.1 冲突场景
针对 docs/conflict.md 文件，模拟了两人对合并流程的不同意见：
- 成员A主张直接合并到 main
- 成员B主张先经过 dev 再合并到 main

### 5.2 冲突产生原因
两人对合并流程有不同意见，导致同一位置内容不同。

### 5.3 解决过程
1. 识别冲突：发现同一文件同一位置有不同修改
2. 协商讨论：团队讨论两种方案的优缺点
3. 达成共识：决定采用先合并到 dev 再合并到 main 的方式
4. 手工解决：编辑冲突文件，保留最终决定
5. 提交合并：完成冲突解决并提交

### 5.4 最终结果
决定采用先合并到 dev 再合并到 main 的方式，保证代码质量和稳定性。
